## 1. 群聊模式 (group)

### 适用场景

- 需要多领域专家从各自角度依次发表意见的场景。
- 头脑风暴、方案评审、多视角讨论。
- 任务不需要严格的先后依赖，但希望每个人都能看到前面的发言并补充。

### 执行流程

```
用户输入任务
      │
      ▼
按团队中 Agent 的顺序排列发言队列
      │
      ▼
┌─────────────────────────────────────────┐
│  for each agent in 队列:                │
│    1. 读取共享白板（含前面所有人的发言） │
│    2. 构造 prompt（任务 + 历史发言）     │
│    3. 调用 LLM 生成发言                 │
│    4. 发言写入共享白板                  │
│    5. 通过 SSE 推送 agent_message 事件  │
│    6. 记录 AgentTrace                   │
└─────────────────────────────────────────┘
      │
      ▼
所有 Agent 发言完毕，汇总最终结果
```

### 借鉴的框架

借鉴 **AG2 (AutoGen) GroupChat** 的顺序发言机制，但简化为固定顺序，不引入智能选择发言者（那是共识模式的能力）。

### 示例

**团队**：产品经理 + UI 设计师 + 前端工程师 + 后端工程师
**任务**：讨论一个登录页的实现方案

```
产品经理：明确登录页需要支持账密、手机验证码、第三方登录三种方式……
UI 设计师：基于产品需求，建议采用卡片式布局，主色调用品牌色……
前端工程师：前端用 React + Tailwind 实现，表单校验用 zod……
后端工程师：后端提供 /auth/login、/auth/sms、/auth/oauth 三个接口……
```

## 2. 流水线模式 (pipeline)

### 适用场景

- 任务有明确的阶段划分，前一阶段的产出是后一阶段的输入。
- 内容生产流水线：选题 → 撰写 → 编辑 → 校对 → 排版。
- 软件开发流水线：需求 → 设计 → 编码 → 测试。

### 执行流程

```
用户输入任务
      │
      ▼
Agent A 接收原始任务
      │
      ▼
产出中间产物 → 写入白板
      │
      ▼
Agent B 读取白板中 A 的产物，作为输入
      │
      ▼
产出中间产物 → 写入白板
      │
      ▼
Agent C 读取白板中 B 的产物，作为输入
      │
      ▼
……接力直到最后一个 Agent
      │
      ▼
最终 Agent 产出最终交付物
```

每个 Agent 只读取上一个 Agent 的产物（而非全部历史），保证流水线的纯粹性。

### 借鉴的框架

借鉴 **CrewAI ProcessType (sequential)** 的顺序流程模式，强调接力与产物传递。

### 示例

**团队**：选题策划 + 文案撰写 + 编辑校对
**任务**：产出一篇技术博客

```
选题策划：基于「多智能体协作」主题，确定文章大纲：1. 背景 2. 框架对比 3. 实战……
文案撰写：（接收大纲）撰写完整正文，约 3000 字……
编辑校对：（接收正文）修正错别字、优化段落衔接、补充摘要……
```

## 3. 管家模式 (master)

### 适用场景

- 任务复杂，需要先拆解再分派给不同领域专家。
- 用户希望有一个「主管」统一把控质量与进度。
- 子任务之间相对独立，可并行或顺序委派。

### 执行流程

```
用户输入任务
      │
      ▼
主管 Agent (Master) 接收任务
      │
      ▼
主管拆解任务为子任务列表
      │
      ▼
┌─────────────────────────────────────────┐
│  for each 子任务:                        │
│    1. 主管选择合适的专家 Agent           │
│    2. 委派子任务（仅 allow_delegation    │
│       的 Agent 可被委派）                │
│    3. 专家执行子任务，产物写回白板       │
│    4. 主管审阅产物                       │
└─────────────────────────────────────────┘
      │
      ▼
主管汇总所有子任务产物 → 最终交付物
```

### 借鉴的框架

借鉴 **CrewAI Hierarchical Process**，由一个主管 Agent 充当协调者，负责任务拆解、委派与汇总。

### 示例

**团队**：项目经理（主管）+ 后端工程师 + 前端工程师 + 测试工程师
**任务**：开发一个待办事项 API

```
项目经理：拆解为 3 个子任务：①后端 API ②前端页面 ③测试用例
项目经理 → 委派 ① 给后端工程师
后端工程师：实现 GET/POST/PUT/DELETE /todos 接口……
项目经理 → 委派 ② 给前端工程师
前端工程师：实现待办列表组件，对接后端接口……
项目经理 → 委派 ③ 给测试工程师
测试工程师：编写 12 条 pytest 用例，覆盖增删改查……
项目经理：汇总，最终交付一个可运行的待办应用。
```

## 4. 共识模式 (consensus)

### 适用场景

- 需要多方达成一致意见的决策场景。
- 方案评审、技术选型、风险评估。
- 希望通过讨论收敛到一个共识结论，而非简单罗列各自观点。

### 执行流程

```
用户输入任务
      │
      ▼
所有 Agent 各自给出初步方案 → 写入白板
      │
      ▼
┌─────────────────────────────────────────┐
│  GroupChatManager 循环:                  │
│    1. 根据当前讨论状态，智能选择下一个    │
│       最合适的发言者（借鉴 AG2）          │
│    2. 被选中的 Agent 读取全部历史，       │
│       发表观点 / 质疑 / 补充              │
│    3. 判断是否达成共识：                  │
│       - 是 → 退出循环                    │
│       - 否 → 继续选择下一个发言者         │
│    4. 设置最大轮次上限，防止无限讨论      │
└─────────────────────────────────────────┘
      │
      ▼
达成共识 → 输出最终结论
```

### 借鉴的框架

借鉴 **AG2 (AutoGen) GroupChat**，核心是 `GroupChatManager` 智能选择发言者机制——根据上下文动态决定下一个该说话的 Agent，而非固定顺序。

### 示例

**团队**：架构师 + 安全工程师 + 性能工程师
**任务**：为新高并发服务选定技术栈

```
架构师：建议用 Go + Redis，吞吐量高……
安全工程师：Go 的生态里安全库不如 Java 成熟，建议考虑 Java + Spring……
性能工程师：Java 的 GC 停顿在高并发下是隐患，Go 更稳……
架构师：可以折中——核心网关用 Go，业务服务用 Java……
安全工程师：同意，Java 侧用 Spring Security……
性能工程师：同意，Go 网关做限流熔断……
[达成共识] 技术栈：Go 网关 + Java 业务服务 + Redis
```

## 5. 并行模式 (parallel)

### 适用场景

- 任务可以拆分为相互独立的子任务，无依赖关系。
- 需要同时从多个角度产出，再合并。
- 追求效率，希望缩短总执行时间。

### 执行流程

```
用户输入任务
      │
      ▼
Orchestrator 接收任务，拆分为 N 个独立子任务
      │
      ▼
┌─────────────────────────────────────────┐
│  并行调度 N 个 Worker Agent:             │
│    Worker A 处理子任务 1 ──┐             │
│    Worker B 处理子任务 2 ──┼── 并行执行  │
│    Worker C 处理子任务 3 ──┘             │
│  各自产出写入白板                        │
└─────────────────────────────────────────┘
      │
      ▼
Orchestrator 收集所有 Worker 产物
      │
      ▼
合并 / 去重 / 整合 → 最终交付物
```

### 借鉴的框架

借鉴 **Anthropic 多智能体架构中的 Orchestrator-Workers 模式**：一个 Orchestrator 负责拆分与汇总，多个 Worker 并行执行独立子任务。

### 示例

**团队**：Orchestrator + iOS 工程师 + Android 工程师 + Web 工程师
**任务**：为一个 API 编写三端 SDK 示例

```
Orchestrator：拆分为 iOS / Android / Web 三个独立示例任务
  ├─ iOS 工程师：（并行）用 Swift 编写 iOS 调用示例……
  ├─ Android 工程师：（并行）用 Kotlin 编写 Android 调用示例……
  └─ Web 工程师：（并行）用 TypeScript 编写 Web 调用示例……
Orchestrator：合并三端示例，统一 README，最终交付。
```

## 6. 转交模式 (handoff)

### 适用场景

- 任务的处理过程中需要动态切换到更合适的专家。
- 客服 / 咨询类场景：通用助手遇到专业问题转给专家。
- 不预先确定执行顺序，由 Agent 自主决定转交。

### 执行流程

```
用户输入任务
      │
      ▼
初始 Agent (如通用助手) 接收任务
      │
      ▼
┌─────────────────────────────────────────┐
│  当前 Agent 处理:                        │
│    1. 读取白板上下文                     │
│    2. 调用 LLM 处理                      │
│    3. LLM 判断:                          │
│       - 自己能处理 → 继续产出            │
│       - 需要转交 → 返回 handoff 指令，   │
│         指定目标 Agent 与转交原因        │
│    4. 若 handoff:                        │
│       - 控制权转移给目标 Agent           │
│       - 发送 agent_handoff SSE 事件      │
│       - 目标 Agent 成为新的当前 Agent    │
│       - 循环回到步骤 1                   │
│    5. 设置最大转交次数上限，防止循环     │
└─────────────────────────────────────────┘
      │
      ▼
某个 Agent 完成处理，不再转交 → 最终产物
```

### 借鉴的框架

借鉴 **OpenAI Swarm** 的 Handoff 机制：Agent 在处理过程中可以主动将控制权转交给另一个 Agent，转交时携带上下文。

### 示例

**团队**：通用助手 + 法律顾问 + 财务顾问
**任务**：咨询公司注册的流程与税务问题

```
通用助手：公司注册流程是：核名 → 提交材料 → 领取执照 → 刻章……
         关于税务问题，我转交给财务顾问。
         [handoff → 财务顾问，原因：涉及税务专业问题]
财务顾问：注册后需在 30 日内税务登记，小规模纳税人……
         关于股权结构的法律风险，我转交给法律顾问。
         [handoff → 法律顾问，原因：涉及法律专业问题]
法律顾问：股权代持需注意……建议签订书面协议……
         [不再转交，完成]
最终产物：包含注册流程 + 税务建议 + 法律风险提示的完整答复。
```

## 7. 状态图模式 (graph)

### 适用场景

- 任务有复杂的依赖关系，无法用简单线性或并行表达。
- 需要条件分支、循环、汇聚等复杂控制流。
- 长流程任务，需要 Checkpoint 断点续跑能力。

### 执行流程

```
用户输入任务
      │
      ▼
构建 DAG 任务图（节点 = Agent 任务，边 = 依赖关系）
      │
      ▼
┌─────────────────────────────────────────┐
│  图调度器循环:                           │
│    1. 找出所有入度为 0 的就绪节点        │
│    2. 并行执行就绪节点对应的 Agent       │
│    3. 节点完成后:                        │
│       - 产物写入白板                     │
│       - 记录 Checkpoint（可断点续跑）    │
│       - 更新下游节点入度                 │
│    4. 支持条件边：根据节点产出决定       │
│       走哪条分支                         │
│    5. 循环直到所有节点完成               │
└─────────────────────────────────────────┘
      │
      ▼
汇聚终态节点产物 → 最终交付物
```

### 借鉴的框架

借鉴 **LangGraph** 的任务状态机：把任务建模为有向图，节点是 Agent 执行单元，边是状态流转（含条件边），并支持 Checkpoint 持久化以实现断点续跑。

### 示例

**团队**：需求分析师 + 架构师 + 前端工程师 + 后端工程师 + 测试工程师 + 审查员
**任务**：开发一个功能模块

```
        ┌──────────────┐
        │  需求分析师   │
        └──────┬───────┘
               │
        ┌──────▼───────┐
        │   架构师      │
        └───┬──────┬───┘
            │      │
   ┌────────▼┐  ┌──▼────────┐
   │ 前端工程师│  │后端工程师  │   (并行)
   └────────┬─┘  └─┬─────────┘
            │      │
            └──┬───┘
         ┌──────▼───────┐
         │  测试工程师   │
         └──────┬───────┘
                │
         ┌──────▼───────┐
         │   审查员      │
         └──────┬───────┘
                │
         (条件边：审查通过 → 结束；不通过 → 回到架构师修改)
                ▼
            最终产物
```

## 8. 模式选择速查表

| 模式 | 标识 | 适用场景 | 借鉴框架 | 是否并行 | 是否动态调度 |
|------|------|----------|----------|----------|--------------|
| 群聊 | `group` | 多视角依次发言 | AG2 GroupChat | 否 | 否（固定顺序） |
| 流水线 | `pipeline` | 阶段接力产出 | CrewAI Sequential | 否 | 否 |
| 管家 | `master` | 主管拆解委派 | CrewAI Hierarchical | 可并行 | 是（主管委派） |
| 共识 | `consensus` | 讨论达成共识 | AG2 GroupChat | 否 | 是（智能选发言者） |
| 并行 | `parallel` | 独立子任务并行 | Anthropic Orchestrator-Workers | 是 | 是（Orchestrator 拆分） |
| 转交 | `handoff` | 动态转交专家 | OpenAI Swarm | 否 | 是（Agent 自主转交） |
| 状态图 | `graph` | 复杂依赖 DAG | LangGraph | 是 | 是（图调度） |

### 如何选择

- **想让大家依次表态** → 群聊模式
- **任务有清晰的阶段流水线** → 流水线模式
- **需要一个主管统筹拆解** → 管家模式
- **需要讨论到达成共识** → 共识模式
- **子任务相互独立、想提速** → 并行模式
- **处理过程中要切换专家** → 转交模式
- **依赖关系复杂、需要断点续跑** → 状态图模式
